## 指令
1. **读取前期收集的信息**：读取在"./[项目名-版本号]/prd"目录下生成的对应的当前软著设计项目的prd.md和static-example.html文件获取信息。
2. **解析PRD模块物理划分清单**：从prd.md中提取"模块物理划分清单"部分，明确每个模块的目录结构、文件命名和职责划分。
3. **生成完整的源代码文件**：严格按照模块物理划分清单，为每个模块生成独立的完整代码文件（而非代码片段），确保：
   - 每个文件都是完整的、可直接运行的代码单元
   - 文件之间的依赖关系清晰（import/引用路径正确）
   - 包含完整的包声明、类定义、方法实现
4. **保存**：严格按照模块物理划分清单的目录结构保存所有代码文件。
   - 在"./[项目名-版本号]/code"目录下按照PRD中的模块物理划分清单创建完整的项目目录结构
   - 每个代码模块独立保存为对应的源代码文件（.py/.js/.java等，根据技术栈确定）
   - **生成README.md**：在"./[项目名-版本号]/code"目录下生成README.md文件，清晰标注：
     * 项目目录结构说明（与模块物理划分清单一致）
     * 前端代码涉及的文件位置及主要文件清单（标注哪些文件属于前端）
     * 后端代码涉及的文件位置及主要文件清单（标注哪些文件属于后端）
     * 技术栈和依赖说明
5. **交接**：提问用户是否要在此基础上生成第五步软著申请完整材料整理，若"是"，则使用`step5-documents-archive`；若"否"，则提示用户修改代码文件至满意。

## 生成要求
1. 基于已有的前端代码进一步进行设计完善，保持软件的前端ui
2. 确保前后端之间的关联性，如：前端 Interface 定义与后端 DTO 实体类在字段名、数据类型、嵌套结构上完全一致。
3. 重点关注源代码的【表达方式】、【逻辑结构】与【算法流程】。
4. **代码文件完整性要求**：
   - 必须为每个模块生成完整的、可独立运行的源代码文件
   - 每个文件包含完整的包声明、导入语句、类/函数定义、方法实现
   - 文件之间通过import/require正确引用，依赖关系清晰
   - 避免使用"..."或"// 其他代码类似"等省略写法
5. **代码行数要求**：
   - 删除空行后的总代码行数必须达到3000行以上
   - 前后端代码比例约为50%:50%
   - 注释占比在20~30%

6. **后端架构要求**（必须满足）：
   - 必须采用**数据库驱动架构**，禁止硬编码业务数据
   - 包含完整的数据库模型定义（如SQLAlchemy ORM类）
   - 包含数据初始化服务（DataInitializer）用于填充基础数据
   - 业务逻辑从数据库读取数据（规则、配置、推荐内容等）
   - 看起来像真实的生产系统，包含完整的CRUD操作
   - 建议包含的模块：数据库模型、数据初始化、业务服务层、API路由层、认证授权、日志记录、错误处理

7. **README.md生成要求**：
   - 使用清晰的目录树结构展示项目文件组织
   - 明确标注每个文件是前端还是后端代码
   - 列出主要代码文件及其功能说明
   - 提供技术栈、依赖库、运行环境等信息

8. 所有注释和文档说明必须使用简体中文。
9. 主体的代码语言需要与用户需求相同。

## 禁止事项
1. **绝不在未获得用户许可的情况下自行生成用户没有提到的功能**
2. **禁止改变前端代码的ui界面**
