## 指令
1. **读取前期收集的信息**：
    - 当前已经使用 `step2-generate-prd` 时：读取在"./[项目名-版本号]/prd"目录下生成的对应的当前软著设计项目的prd.md文件获取信息。
    - 当前未使用 `step2-generate-prd` 时：询问用户对应的前期确认的prd文件的位置信息，读取对应md文件的内容获取信息。

2. **生成软著的软件使用说明文档**： 生成符合《计算机软件著作权登记办法》要求的格式规范，内容完整的《软件使用说明文档》内容，格式可参考下方的软件说明文档内容结构部分。
    - **文段式叙述**：正文内容需要采用文段式叙述，使用严谨且书面化的语言，禁止过多使用括号补充叙述的情况。
    - **功能截图数**：软件使用说明文档中每一个功能都需要一个对应的功能截图，其他位置禁止生成图片。
    - **IMAGE_MASK规则**：正文内容需要功能截图的位置采用`[IMAGE_MASK]`标记，标记**必须独占一行**。
    - **IMAGE描述**：根据`[IMAGE_MASK]`位置顺序生成每一张功能截图的描述。

3. **静态样例网页生成**：
    - 读取当前项目"./[项目名-版本号]/manual"目录下的"imgdescription.md"，生成可直接打开的符合要求的静态网页样例，用于替换对应位置的`[IMAGE_MASK]`。
    - 你只需要根据需求生成是包含完整功能的软件静态网页，无需考虑截图。

4. **保存**：当对应的内容生成完全时，将整理好的信息保存。
    - 将生成的正文内容保存在当前项目的"./[项目名-版本号]/manual"目录下，命名方式为"manual.md"
    - 将生成的`[IMAGE_MASK]`的功能截图描述内容保存在当前项目的"./[项目名-版本号]/manual"目录下，命名方式为"imgdescription.md"
    - 将生成的前端静态网页内容保存在当前项目的"./[项目名-版本号]/manual"目录下，命名方式为"static-example.html"

5. **间歇策略**：每个任务可以视作一个子环节，每完成一个子环节询问用户是否进行下一步，采用间歇机制，按顺序逐个完成相关任务

6. **交接**：提问用户是否要在此基础上生成软件的代码手册，若"是"，则使用 `reference: step4-generate-code`；若"否"，则打开创建后的md文件供用户修改至满意后使用 `reference: step4-generate-code`。

## 《软件使用说明文档》设计要求

```markdown
# [系统名称][版本号]

**一、需求分析**

在当前[行业/领域]背景下，存在以下问题：[具体问题描述]。传统方式存在[局限性1]、[局限性2]等问题。

本项目旨在通过[解决方案概述]，解决上述痛点，实现[目标1]、[目标2]，从而[预期效果]。

**二、技术背景**

**2.1 [技术A名称]**
[技术A]是[技术定义]，具有[特点1]、[特点2]等优势。在本项目中主要用于[具体用途]，其[某个特性]能够显著提升[某方面性能]。

**2.2 [技术B名称]**
[技术B]是[技术定义]，作为[技术A]的补充/扩展，提供[特定功能]。与[技术A]的集成能够[协同效果]。

**三、功能模块**

**3.1 [模块名称]**
**3.1.1 [子模块名称]**
内容包含模块的功能，用户使用功能模块的方式，模块与用户的交互方式，内容不可设计前后端函数等不该出现在用户手册的内容。

[IMAGE_MASK]

**3.2 [模块名称]**
**3.2.1 [子模块名称]**
内容包含模块的功能，用户使用功能模块的方式，模块与用户的交互方式，内容应当侧重使用，严禁涉及前后端函数等不该出现在用户手册的内容。

[IMAGE_MASK]

**3.2.2 [子模块名称]**
...

```

---
**格式规范：**
1. 使用三个固定主标题：一、需求分析；二、技术背景；三、功能模块
2. 一、需求分析；二、技术背景 部分都不需要图片，图片只需要在三、功能模块处出现
3. 在一个功能模块存在多个子功能时必须划分子模块章节
4. 软著需要的说明文档在功能模块章节应当侧重用户使用方式而非侧重模块的详细实现方式
5. 每个功能模块后必须跟截图说明（使用[IMAGE_MASK]代替）
6. 避免使用"我们""您"等人称代词，使用客观描述
7. 字数范围要求在4000~5000字
